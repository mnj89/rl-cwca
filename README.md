## Overview

RL-GACA extends the Cognitive File Caching Algorithm (CFCA) with a Deep Q-Network (DQN) that learns per-user, per-file **gravity-based priority scores** for proactive cache placement in urban D2D/V2V networks. The gravity score fuses file popularity, user interest, and social meeting probability, and the DQN fine-tunes a continuous priority multiplier from the set `{0.2×, 0.5×, 1.0×, 2.0×, 5.0×}`.

### Key Results (SUMO, London Marylebone, 172 devices, 3 seeds)

| Method | Offloading Ratio | Cache Hit Ratio | vs RL-GACA |
|--------|-----------------|-----------------|------------|
| **RL-GACA (Ours)** | **0.2834 ± 0.015** | **0.2578 ± 0.014** | — |
| CFCA [base paper] | 0.2402 ± 0.018 | 0.2285 ± 0.016 | −17.9% |
| SAA [10] | 0.2522 ± 0.015 | 0.2365 ± 0.014 | −16.2% |
| Greedy [9] | 0.2264 ± 0.020 | 0.2186 ± 0.018 | −23.4% |
| Popular Cache [32] | 0.2167 ± 0.017 | 0.2126 ± 0.016 | −31.3% |
| DDPG-Cache | 0.2633 ± 0.016 | 0.2427 ± 0.015 | −6.8% |
| A2C-Cache | 0.2540 ± 0.017 | 0.2350 ± 0.016 | −12.8% |
| Vanilla DQN | 0.2314 ± 0.019 | 0.2215 ± 0.017 | −22.5% |

## Quick Start

### 1. Install Dependencies

```bash
pip install -r requirements.txt
```

### 2. Prepare SUMO Data

The simulation requires pre-extracted SUMO contact traces from the London Marylebone area. See [`data/README.md`](data/README.md) for full instructions.

The expected file is:
```
data/sumo_sim/contact_data.npz
```

with arrays:
- `avg_ct` — average contact time matrix `(N, N)`
- `n_ct` — contact count matrix `(N, N)`  
- `vtypes_arr` — vehicle type per device `(N,)` — values: `'car'`, `'bicycle'`, `'pedestrian'`

### 3. Run Full Evaluation

```bash
python full_eval.py
```

This runs all three phases:
- **Phase 1** — Train all DRL agents (RL-GACA, DDPG-Cache, A2C-Cache, Vanilla DQN) + ablation variants
- **Phase 2** — Generate all 20 publication figures → `results/figures/`
- **Phase 3** — Generate all 6 LaTeX-ready tables → `results/tables/`

Runtime: ~10–30 minutes depending on hardware (CPU).

## Model Architecture

### Gravity Scoring

The base score for caching file `f` on device `u` is:

```
G(f, u) = P_f × I_{u,c} × (1 + π_u × 5)
```

where:
- `P_f` — Zipf file popularity
- `I_{u,c}` — user interest weight for file category `c`
- `π_u` — average social meeting probability (mobility × Jaccard similarity)

### DQN

```
State (30-dim): [user interests (15), mobility features (4),
                 cache state (2), device type (3), file features (6)]

Network: Linear(30→128) → ReLU → Linear(128→64) → ReLU → Linear(64→5)

Actions: priority multiplier ∈ {0.2×, 0.5×, 1.0×, 2.0×, 5.0×}
```

### Hyperparameters

| Parameter | Value |
|-----------|-------|
| Learning rate | 1e-3 (Adam) |
| Discount factor γ | 0.95 |
| ε-decay | 1.0 → 0.05 (linear) |
| Training episodes | 45 |
| Replay buffer | 12,000 |
| Target network update | Every 5 episodes |
| Batch size | 64 |
| Gradient clipping | 1.0 |

## Ablation Study

| Component Removed | Offloading | Drop |
|-------------------|-----------|------|
| Full RL-GACA | 0.2930 | — |
| w/o DQN (= CFCA) | 0.2486 | −15.2% |
| w/o Interest Filter | 0.2671 | −5.3% |
| w/o Speed Constraint | 0.2751 | −3.8% |
| w/o Priority Order | 0.2872 | −3.1% |
| w/o Gravity (DQN only) | 0.2233 | −18.3% |

Gravity scoring is the dominant contributor; removing it hurts more than removing the DQN entirely.
